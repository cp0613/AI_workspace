# Linux 内核技术调研报告

针对 Linux 内核技术主题生成深度中文调研报告，报告以 Markdown 文件保存到 `<skill_dir>/../../knowledge/gen/research_report/` 目录。

## 触发条件

当用户出现以下情况时激活此技能：
- 要求调研或研究某个内核子系统或机制
- 需要对某个内核特性进行详细技术说明
- 要求分析与某规范或标准相关的内核源码
- 使用关键词："调研"、"研究"、"分析"、"报告"、"技术点"、"原理"

## 工作流程

### 步骤一：明确范围

开始前与用户确认：
1. 具体的调研主题或内核子系统
2. 分析哪个内核版本/分支（默认：当前仓库）
3. 是否有特定的关注点或待回答的问题

### 步骤二：调研

使用 explore-agent 或搜索工具收集信息：

1. **源码分析**：搜索并阅读相关内核源文件
   - 识别关键数据结构（`struct`、`enum`、`union`）
   - 追踪初始化和运行时调用链路
   - 记录重要的宏定义、配置选项（`CONFIG_*`）及 Kconfig 依赖关系
   - 使用 `Grep`、`Glob`、`Read` 工具导航代码库

2. **规范文档引用**：当主题涉及硬件规范时：
   - RISC-V ISA spec、AIA spec、SBI spec 等
   - 引用具体的章节编号和版本
   - 需要时使用 `WebSearch`/`WebFetch` 查阅官方规范文档

3. **跨架构对比**：调研同一技术点在不同架构下的实现方案
   - 主要对比架构：RISC-V（`arch/riscv/`）、ARM64（`arch/arm64/`）、x86（`arch/x86/`），其他架构如有对应实现方案，也需完成该架构的调研
   - 关注点：设计思路差异、硬件机制差异、软件抽象层差异
   - 搜索各架构下对应的源码目录和头文件
   - 搜索 lkml 上有关的讨论过程
   - 搜索 LWN.net 上有关的介绍
   - 搜索 git 提交历史中有关的介绍
   - 如果某架构无对应实现或采用完全不同的机制，也需说明原因

4. **相关文档**：查阅 `Documentation/` 目录下的内核文档

### 步骤三：生成报告

按照 [report-template.md](report-template.md) 中的模板创建报告。

**文件命名规范**：`<英文主题-kebab-case>.md`
- 示例：`riscv-aplic-msi-mode.md`、`cfs-scheduler-overview.md`

**输出路径**：`<skill_dir>/../../knowledge/gen/research_report/<文件名>.md`

### 步骤四：审核

报告完成后：
1. 确认所有源码引用都包含文件路径和行号
2. 确保架构图在 Markdown 中正确渲染
3. 检查规范引用包含章节编号
4. 确认报告回答了用户的原始问题

## 报告质量规范

### 源码分析
- 始终包含文件路径和行号：`[file.c](path/to/file.c:42)`
- 展示关键代码片段（保持简短，5-15 行）
- 解释代码背后的"为什么"，而不仅仅是"做了什么"
- 追踪调用链：`caller() -> callee() -> ...`

### 架构图示
- 使用 ASCII 艺术字或文本图表以保证兼容性
- 使用方框字符提高清晰度：
  ```
  +------------------+
  |     组件 A       |
  +--------+---------+
           |
           v
  +--------+---------+
  |     组件 B       |
  +------------------+
  ```
- 标注所有箭头和连接关系
- 保持图表聚焦（每张图一个概念）

### 规范文档引用
- 格式：`[规范名称] Section X.Y.Z (vN.N)`
- 示例：`[RISC-V AIA Spec] Section 4.5 (v1.0)`
- 引用规范中的关键定义

### 方案对比
- 使用表格进行并排对比
- 包含维度：依赖项、性能、复杂度、适用场景、局限性
- 清晰陈述各方案的权衡取舍

### 跨架构对比
- 以 RISC-V 实现为主线，对照 ARM64 和 x86 的实现方案，其他架构如有实现方案也加入对比
- 对比维度：硬件机制、软件抽象、关键数据结构、中断/异常处理流程等
- 源码引用需标注各架构的文件路径，例如：
  - RISC-V: `arch/riscv/kernel/traps.c`
  - ARM64: `arch/arm64/kernel/entry-common.c`
  - x86: `arch/x86/kernel/traps.c`
- 若某架构无对应概念或采用完全不同的机制，需说明根本原因（硬件差异、规范差异等）
- 总结各架构设计的优劣和可借鉴之处

### 使用方法
- 涵盖从启用配置到实际使用的完整链路
- 区分内核态接口和用户态接口
- 提供可直接参考的代码示例和命令示例
- 包含设备树配置、内核启动参数等实际部署信息（如适用）
- 列出调试和诊断手段，帮助读者快速定位问题

## 语言和风格

- 报告正文使用中文
- 技术术语首次出现时保留英文原文并附中文解释
  - 示例："APLIC (Advanced Platform Level Interrupt Controller, 高级平台级中断控制器)"
- 代码标识符、文件路径、寄存器名保持英文
- 章节标题使用中文
