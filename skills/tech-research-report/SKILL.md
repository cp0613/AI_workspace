# 特定技术调研报告

针对特定技术主题生成深度中文调研报告，报告以 Markdown 文件保存到 `<skill_dir>/../../knowledge/gen/research_report/` 目录。

## 触发条件

当用户出现以下情况时激活此技能：
- 要求调研或研究某个非内核的技术主题（内核主题优先使用 kernel-research-report）
- 需要了解某个工具、框架、协议或规范的原理和使用方法
- 使用关键词："调研"、"研究"、"分析"、"报告"、"怎么用"、"原理"、"选型"

## 工作流程

### 步骤一：明确范围

开始前与用户确认：
1. 具体的调研主题
2. 分析的目标版本（如有版本概念）
3. 是否有特定关注点（原理深度、使用方法等）
4. 目标受众和应用场景

### 步骤二：调研

使用 explore-agent 和搜索工具收集信息：

1. **官方资料收集**：
   - 使用 `WebSearch`/`WebFetch` 查找官方文档和规范
   - 定位官方 GitHub 仓库和参考实现
   - 收集官方 spec/标准文档的链接和版本信息
   - 查找官方推荐的工具和配套项目

2. **原理分析**：
   - 理解核心架构和设计思路
   - 识别关键数据结构、接口和协议
   - 梳理工作流程和调用链路
   - 分析设计决策背后的原因

3. **源码/实现分析**（如适用）：
   - 阅读本地仓库中的相关代码
   - 阅读 GitHub 上的参考实现
   - 识别关键模块和入口点
   - 使用 `Grep`、`Glob`、`Read` 工具导航代码库

4. **生态和工具调研**：
   - 调研相关的开发工具、调试工具
   - 查找社区常用的辅助库和框架
   - 了解 CI/CD、测试等配套方案

5. **社区资源**：
   - 搜索 LWN.net、邮件列表上的相关讨论
   - 查找技术博客和会议演讲
   - 搜索 git 提交历史中的相关信息

### 步骤三：生成报告

按照 [report-template.md](report-template.md) 中的模板创建报告。

**文件命名规范**：`<英文主题-kebab-case>.md`
- 示例：`qemu-virtio-device-emulation.md`、`riscv-aia-spec-overview.md`

**输出路径**：`<skill_dir>/../../knowledge/gen/research_report/<文件名>.md`

### 步骤四：审核

报告完成后：
1. 确认所有官方 spec 链接有效且标注了版本
2. 确认 GitHub 参考代码链接准确
3. 确保使用方法章节包含可操作的步骤
4. 检查工具推荐包含安装和基本用法
5. 确认报告回答了用户的原始问题

## 报告质量规范

### 官方规范引用
- 必须包含官方 spec/标准文档的完整链接
- 标注规范版本号和具体章节：`[规范名称] Section X.Y.Z (vN.N)`
- 引用规范中的关键定义和术语

### GitHub 参考实现
- 列出官方和社区的 GitHub 参考仓库链接
- 指出仓库中的关键目录和文件
- 标注仓库的活跃度和维护状态
- 如有本地代码，使用 `[file.c](path/to/file.c:42)` 格式引用

### 原理分析
- 使用 ASCII 架构图展示组件关系
- 展示关键代码片段（5-15 行）
- 解释"为什么这样设计"，而不仅仅描述"做了什么"
- 追踪调用链：`caller() -> callee() -> ...`

### 使用方法
- 涵盖从安装/启用到实际使用的完整链路
- 提供可直接复制执行的命令和代码示例
- 区分不同使用场景（开发、调试、生产）
- 包含常见问题和排查方法

### 工具与生态
- 列出相关工具并附官方链接
- 提供安装命令和基本使用示例
- 标注工具的适用场景和局限性

### 方案对比（如适用）
- 使用表格进行并排对比
- 包含维度：原理、性能、复杂度、适用场景、局限性
- 清晰陈述各方案的权衡取舍

## 语言和风格

- 报告正文使用中文
- 技术术语首次出现时保留英文原文并附中文解释
  - 示例："Virtio (Virtual I/O, 虚拟 I/O 框架)"
- 代码标识符、文件路径、命令、URL 保持英文
- 章节标题使用中文
